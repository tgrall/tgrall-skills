# Interview guide

Use this guide to ask focused questions for each PRD block. You do not need to ask every question verbatim. Use the smallest set of questions that gets the user to a complete, concrete answer.

## 0. Language setup

Start here before gathering product requirements.

Typical questions:

- Which language should I use for the conversation?
- Which language should I use for the PRD itself?
- Would you like to use the system default, or choose one explicitly?
- Suggested options: English, French, German, Italian, Spanish, Dutch, or another language you name.

Completion check:

- The conversation language and PRD language are known.
- If the user does not choose, the system default is treated as the choice.

## 1. Metadata table

Capture the document identity.

Typical questions:

- What is the product or feature name?
- Is this a net-new product, a major feature, or an iteration on something existing?
- What version or milestone should this PRD represent?
- Who should be listed as author or owner?
- What draft status should be shown?
- What language should be recorded for the PRD?

Completion check:

- The document can be uniquely identified and attributed.
- The PRD language is reflected in the metadata.

## 2. Executive Summary

Capture the short narrative overview.

Typical questions:

- In one or two sentences, what is this product or feature?
- Who is it for?
- What core value does it create?
- What makes this approach worth building now?

Completion check:

- A stakeholder can read the summary and understand the idea without reading the rest of the document.

## 3. Problem Statement

Define the pain clearly and specifically.

Typical questions:

- What is broken, slow, confusing, manual, risky, or missing today?
- Who feels this problem most acutely?
- What do people do today instead?
- Why are current tools or workflows insufficient?
- What happens if this problem remains unsolved?

Completion check:

- The problem is concrete, user-centered, and tied to a meaningful consequence.

## 4. Goals & Objectives

Define success in observable terms.

Typical questions:

- What outcomes should this initiative achieve?
- How will we know it is working?
- Are there success metrics, adoption signals, quality thresholds, or time-to-value targets?
- Are there business, user, or operational goals that matter most?

Completion check:

- Goals are explicit and at least partially measurable or otherwise testable.

## 5. Target Audience

Identify who the PRD is for.

Typical questions:

- Who are the primary users or personas?
- Which user group matters most for the first version?
- What job are they trying to get done?
- What constraints or behaviors define them?

Completion check:

- The document identifies concrete user types and their primary needs.

## 6. User Stories

Build the functional core of the PRD.

Typical questions:

- What are the main user workflows?
- What must a user be able to do from start to finish?
- What supporting actions are needed around creation, editing, organization, discovery, deletion, permissions, sharing, reporting, or settings?
- Which stories are must-have versus should/could-have?
- What acceptance criteria define success for each story?
- Are there important edge cases, recovery flows, or non-happy paths?

Completion check:

- Stories cover the key workflows, not just one happy path.
- Each story has a clear benefit and acceptance criteria.
- Priorities are assigned.

## 7. Prioritization

Turn the raw story list into a delivery shape.

Typical questions:

- What is absolutely required for an MVP or first release?
- What can wait until a later phase?
- Which features are valuable but not essential?
- Are there any explicitly rejected items for this version?

Completion check:

- The PRD clearly separates must, should, could, and won't-have scope, or uses another explicit prioritization model.

## 8. Out of Scope

Set boundaries to prevent scope creep.

Typical questions:

- What related ideas are intentionally excluded?
- What might people assume is included, but is not?
- Which later-phase capabilities should be called out explicitly as out of scope for now?

Completion check:

- A reader can tell where this initiative stops.

## Optional follow-up blocks

Use these when they materially improve the PRD:

- assumptions
- risks
- dependencies
- implementation notes
- release phases
- open questions

Ask for these especially when the user wants engineering alignment, staged delivery, or a stronger planning artifact.
